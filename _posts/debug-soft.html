软件调试

软件调试在软件开发中占有很大比重，有时候调试比写代码占用了更多的时间。软件调试的目的多是为了确保业务流程正确、发现软件的性能瓶颈、定位并解决软件的Bug等，本文主要想讲的是定位Bug。

直觉和想象

当遇到一个Bug的时候，依靠自己的直觉和想象定位可能出现Bug的原因，对可能造成Bug的代码，在脑海中进行静态分析，找出Bug的根因并给出解决方案。

这个方算是一个神级操作，看起来什么都没有做就把问题解决了。这需要你对软件代码足够的熟悉，能在脑海中模拟软件的执行过程。但是这只能来解决你自己编写代码，或者你深度阅读过的代码。脑海中对的代码进行静态分析的能力，可以通过大量阅读代码，大脑持续模拟执行来加强。

Code Review

打开代码结合Bug现象，确定Bug的原因。如果早期进行了比较好的Code Review，在软件发布后就省下很多时间。

断点调试

断点调试可以一步一步的查看软件的执行过程，直接定位到Bug的位置。

断点调试是比较常用的调试方式，十分有效。但是也有不少缺点：基本上只能在开发环境上断点调试；调试的上下文是一次性的，不能保存，如果需要只能进行二次调试；当项目较大时断点调试可能需要花费大量的时间。

日志调试

日志记录下软件的执行过程，能直接记录Bug发生的原因，或者可以供程序员进行分析。

日志记录在开发环境和生产环境同样有效。日志记录的是持久化内容，可以多次查看、共享等。跟好的日志可以通过软件读取日志记录的数据，重新浮现Bug,可结合断点调试找到产生Bug的原因。

调试工具

调试工具可以用不编程的方式解决软件Bug。

  • CPU
    • 查看软件中哪些代码消耗较多的CPU，一个模块的具体函数在所监测时间段内执行次数、占用的CPU百分比、时钟周期等。对占用CPU较多的代码进行优化、重构、重写，更加有效地使用CPU。
  • 内存
  • 数据访问
  • 系统工具
